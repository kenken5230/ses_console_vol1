# Gmail取り込み 実装状況メモ v0.1

作成日: 2026-05-11

このメモは、Gmail OAuth認証、Gmail取得、`mail_notifications` 保存、ルール分類、案件/要員抽出、未分類タブ実装後の現状を整理するものです。

## 1. 確定した前提

- `ses@skv.co.jp` はGoogleグループであり、OAuth認証対象ではない。
- OAuth認証する実体アカウントは `sho.sato@skv.co.jp`。
- `sho.sato@skv.co.jp` はGoogle Workspace管理者ではない。
- Admin SDK / Directory API / ドメインワイドデリゲーションは前提にしない。
- Gmail APIは `userId = "me"` で `sho.sato@skv.co.jp` のメールボックスを読む。
- 初期検索クエリは `to:ses@skv.co.jp`。
- `deliveredto:ses@skv.co.jp` は実測でヒットしなかったため、初期実装では使わない。
- Gmailスコープは `https://www.googleapis.com/auth/gmail.readonly`。
- Gmail push通知、Pub/Sub、ラベル操作、既読化、メール送信、添付保存はまだ行わない。
- メールは保存期限で自動削除しない。

## 3. 同期処理の現状

当初は最新50件程度の手動同期を想定していたが、実運用では3月以降のメールをすべて保持する方針に変更した。

現在の同期処理:

- Gmail `users.messages.list` の `nextPageToken` を使ってページング取得する。
- 既定の同期開始日は `2026-03-01`。
- 既定のページサイズは最大 `500`。
- `--from=YYYY-MM-DD`、`--to=YYYY-MM-DD`、`--limit=N`、`--page-size=N`、`--query=...` を指定できる。
- 既存メールは `sourceAccountId + externalMessageId` で判定し、通常は詳細再取得をスキップする。
- 既存メールを更新したい場合のみ `--refresh-existing` を使う。
- メール保存時に物理削除や保存期限による削除は行わない。

確認済みの同期結果:

|---|---:|
| 対象期間 | 2026-03-01以降 |

| 最古 `receivedAt` | `2026-03-01T15:00:32.000Z` |
| 最新 `receivedAt` | `2026-05-10T05:30:20.000Z` |

注記:
Gmail検索の日付演算子はタイムゾーン境界に注意が必要。3月1日JSTの早い時間帯にメールが存在するはずなのに取得されない場合は、検証用に `--from=2026-02-28` で境界確認する。

## 6. 重複判定方針

### 6.1 メール原本の保存

`mail_notifications` の保存重複判定は、引き続き以下を使う。

- `sourceAccountId`
- Gmail APIの `message.id` を保存した `externalMessageId`

理由:

- 同じ会社・同じ件名でも、Gmail上では別メールとして届くことがある。
- 原本メールは監査・確認のために全件保持したい。
- そのため、`mail_notifications` では同じ会社・同じ件名を理由に保存スキップしない。

### 6.2 案件/要員への抽出

`projects` / `persons` への抽出では、同一内容の営業メールが複数回届くことで案件/要員が増殖しやすい。

そのため抽出時には、追加で以下を重複候補として扱う。

- 同じ送信元ドメイン
- 同じ `normalizedSubject`

この条件に一致する既存の案件/要員がある場合、新規作成せず、対象メールを既存エンティティへ `mail_entity_links` / `extraction_results` で紐付ける。

## 8. 未分類タブの現状

メインタブは以下。

- 案件
- 要員
- 未分類

未分類タブには、案件/要員として抽出されていない確認対象メールを表示する。
トップ画面にメール通知一覧カード、提案一覧、配信履歴は復活させない。

未分類タブの主な対象:

- `category` が `NEEDS_REVIEW` / `OTHER` / `NORMAL_CONTACT` 相当
- `isExcluded = false`
- 有効な `sourceProjects` / `sourcePersons` がない

未分類メールのサイドペインでは、本文全文を確認し、必要に応じて以下の操作ができる。

- 案件として扱う
- 要員として扱う

この操作では、対象メールの `category` を更新し、既存の抽出ロジックを使って案件/要員へ仮登録する。
同じメールから同じproject/personを重複作成しない。

## 12. サーバー運用時の同期・更新方針

SES Consoleは、将来的にサーバーへ配置し、複数人が同時に利用する前提にする。
そのため、Gmail同期や一覧更新は「業務中の画面を勝手に変えない」ことを最優先にする。

### 12.1 15分おきのサーバー側自動同期

- Gmail同期はサーバー側ジョブとして15分おきに実行する。
- 同期対象は `GMAIL_QUERY=to:ses@skv.co.jp` を基本にする。
- `mail_notifications` への保存は `sourceAccountId + externalMessageId` で重複防止する。
- 同期ジョブ内で、保存、ルール分類、案件/要員の仮抽出まで実行する方針にする。
- 画面を開いているユーザーの一覧は、自動同期完了時に勝手に差し替えない。
- 同期結果は別途ログとして残し、何件取得、何件分類、何件抽出、何件失敗したか確認できるようにする。

### 12.2 手動同期

- 画面に手動同期ボタンを用意する。
- 手動同期は、管理者または同期権限を持つユーザーのみ実行できるようにする。
- 手動同期実行中は、二重実行を防ぐためボタンをdisabledにする。
- 同期完了後も一覧は勝手に更新しない。
- 完了時は「新着データがあります」「同期が完了しました。表示を更新しますか？」のように通知する。
- ユーザーが明示的に更新操作をした場合のみ、一覧データを再取得する。

### 12.3 同期と画面更新を分離する

Gmail同期はDBを更新する処理であり、画面表示の更新とは分けて扱う。

- バックグラウンド同期: DBへ新着メール、分類結果、抽出結果を保存する。
- 画面通知: 新着件数や更新可能な状態だけを表示する。
- 画面反映: ユーザーが更新ボタンを押した時だけ一覧へ反映する。

これにより、編集中や確認中の画面が突然切り替わることを防ぐ。

### 12.4 新着通知

- 自動同期後、画面には「新着データがあります」と表示する。
- 可能であれば、新着件数を分類別に表示する。
  - 新着メール
  - 案件候補
  - 要員候補
  - 未分類
- 通知は一覧の内容を勝手に差し替えず、ユーザーの判断で更新できる導線にする。
- サイドペインや作成/編集フォームを開いている場合も、入力内容を壊さない。

### 12.5 同時編集対策

複数人利用時は、同じ案件/要員を同時に編集する可能性がある。
業務影響を減らすため、以下を検討する。

- 編集開始時に編集ロックを作成する。
- 他ユーザーが同じ案件/要員を開いた場合、「○○さんが編集中です」とリアルタイムまたは準リアルタイムで表示する。
- ロックは一定時間操作がない場合に自動失効する。
- 編集者本人が閉じる、保存する、キャンセルする場合はロックを解除する。
- 保存時には `updatedAt` または将来的な `version` カラムを使って競合を確認する。
- 他ユーザーが先に保存していた場合は、上書きせず「他のユーザーが更新しています」と警告する。
- 初期実装ではDBスキーマ変更なしで `updatedAt` による保存時競合チェックから始める。
- 本格実装では `entity_edit_locks` のような編集ロックテーブルを追加する案を検討する。

### 12.6 アカウントごとのログイン・権限管理

サーバー運用時は、ユーザーごとにログインして操作する前提にする。

- `users` をログインユーザーの基盤として扱う。
- 作成者、更新者、同期実行者を記録できるようにする。
- 将来的に権限を分ける。
  - 管理者
  - 営業
  - 閲覧のみ
  - 同期実行権限あり
- 重要操作は `audit_logs` に記録する。
- 手動同期、案件作成、案件編集、要員作成、要員編集、未分類からの移動は、誰が実行したか追えるようにする。

### 12.7 実装順序

1. Gmail同期ジョブの15分実行設計
2. 手動同期API/ボタン
3. 同期結果テーブルまたは同期ログ
4. 画面の「新着あり」通知
5. 編集ロック設計
6. 保存時の競合チェック

## 13. 案件/要員分類ルール精査方針

3月1日から5月10日までに取り込んだメールを対象に、案件紹介と要員紹介をできるだけ正確に区別できるルールを作る。
ルール作りには時間をかけてよい。
まずはAI分類ではなく、分析、仮説、ルール化、再分類、確認のサイクルで精度を上げる。

### 13.1 目的

- 未分類タブに残るメールを減らす。
- 案件メールを案件タブに、要員メールを要員タブに自然に出せるようにする。
- 「要員募集」と「要員紹介」のような紛らわしい表現を誤分類しない。
- ルールの根拠を残し、後から調整できるようにする。
- 分類結果は人間が確認・修正できる前提にする。

### 13.2 分析対象

- 対象期間: 2026-03-01 から 2026-05-10
- 対象テーブル: `mail_notifications`

  - `bodyText`
  - `normalizedBody`

  - `fromName`
  - `toEmails`

### 13.7 ルール作成の手順

1. 3月1日から5月10日までの全メールを対象に、現状分類を集計する。
2. `OTHER` / `NEEDS_REVIEW` を送信元、件名、本文キーワードでクラスタリングする。
3. 未分類上位から、人間が案件/要員/除外/通常連絡にラベルを付ける。
4. 要員側の強いルールを先に作る。
5. 案件側の強いルールを作る。
6. 紛らわしい表現の優先順位を決める。
7. ルールを `scripts/gmail-classification-rules.ts` に反映する。
8. `gmail:classify` を再実行する。
9. 分類結果のサマリを確認する。
10. 未分類件数、誤分類例、要確認件数を見て再調整する。

### 13.8 ルール評価の観点

- 未分類件数が減っているか。
- 要員紹介が案件に流れていないか。
- 案件紹介が要員に流れていないか。
- セミナー、メルマガ、営業広告が通常一覧に混ざっていないか。
- `案件探してます` と `要員募集` のような逆向き表現を誤判定していないか。
- 同じ送信元の定型メールが安定して同じ分類になるか。
- 分類後に `gmail:extract` したとき、案件/要員タブの見え方が業務上自然か。

### 13.9 成果物

- UIが崩れた状態で分類作業を進めると、分類不具合なのか画面不具合なのか判断しづらくなる。
