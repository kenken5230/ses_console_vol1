# AI作業ルール更新 引き継ぎメモ 2026-06-22

## 目的

`1タスク1サブエージェント` の意味を、逐次処理ではなく、独立した作業パッケージごとの担当分離として明確化した。

同時並行しやすいタスク粒度を定義し、細かすぎる分割による待機、引き継ぎ、整合確認コストを抑える。

## 体制

- 親PM: 方針統合、最終反映、確認
- 監査サブ: 並列化の安全条件、NG条件、監査ポイントを確認
- 監視サブ(PMO): タスク粒度、並列キュー、記録項目を確認
- テクニカルリード: write scope分離、競合回避、統合チェックを確認
- 実行者: 長文版と短縮版の文面案を作成

## 変更ファイル

- `AI_WORK_RULES.md`
- `AI_WORK_RULES_SHORT.md`

## 主な変更

- `サブエージェント並列実行・タスク粒度ルール` を長文版に追加
- 短縮版へ同方針を要約して追加
- `1タスク1サブエージェント` は逐次処理ではなく、独立作業パッケージに1サブを割り当てる意味だと明記
- 依存関係がなくwrite scopeが分離されたタスクは同時並行で進める方針を追加
- 1タスクの目安を30〜90分程度に設定
- 細かすぎるタスク分割と大きすぎるタスクの分割基準を追加
- write scope分離、並列禁止条件、並列キュー分類、タスク記録項目、統合時チェックを追加

## 監査結果

- 条件付きOK
- 同じファイル、同じAPI契約、同じDB/schema、同じ状態管理を複数サブが同時編集しないこと
- DB/schema/env/package/lockfile、削除、merge、Ready化、close、deployなど重要操作は並列実行せず既存安全ルールを優先すること
- dirty workspaceではユーザー作業を巻き込まないこと
- 親PMが並列投入前に依存関係、write scope、完了条件、検証方法を整理すること

## 残リスク

- タスク粒度30〜90分は目安であり、実作業で調整が必要
- 複数サブが同じ領域を触る場合は、親PMまたはテクニカルリードが直列化する必要がある

## 次に見るべきこと

- タスクが細かすぎる作業手順になっていないか
- 同時並行できるREADYタスクが待たされていないか
- write scopeが重複していないか
- 統合前にchanged files、deleted files、未実行検証、rollback方針が記録されているか
