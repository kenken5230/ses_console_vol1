# AI作業ルール更新 引き継ぎメモ 2026-06-21

## 目的

ユーザー確認を最小化しつつ、親PMとサブエージェント間の内部確認を強化するルールを追加した。

## 体制

- 親PM: 方針統合、最終反映、確認
- 監査サブ: 安全条件、弱めてはいけない既存ルール、NG表現を確認
- 監視サブ(PMO): 記録項目、判断割れ時の扱い、進行管理を確認
- テクニカルリード: 技術判断を内部完結させる条件、テスト、差分、rollback、仕様変更検知を確認
- 実行者: 長文版と短縮版の文面案を作成

## 変更ファイル

- `AI_WORK_RULES.md`
- `AI_WORK_RULES_SHORT.md`

## 主な変更

- `ユーザー確認最小化ルール` を長文版に追加
- 短縮版へ同方針を要約して追加
- ユーザーを技術作業の承認者ではなく、損失、仕様、事業影響、安全性の承認者として扱う方針を明記
- 通常の実装、修正、検証、docs更新、非破壊差分、明らかなバグ修正は内部確認で進める方針を明記
- 削除、大量削除、データ消失、DB write、仕様変更、本番/共有環境影響、外部公開、課金、秘密情報/個人情報、セキュリティ低下、内部監査NG/判断割れはユーザー確認必須とした
- 重要権限では、監査サブ、監視サブ(PMO)、テクニカルリードのOKを必須にした

## 監査結果

- 条件付きOK
- 内部承認はユーザー承認必須事項の代替ではない
- 判断割れ、NG、保留、未確認、不在がある場合は実行不可
- ユーザー確認不要と判断した理由、changed/deleted files、仕様影響、DB/schema/env/package/lockfile変更、未実行検証、rollbackを記録すること

## 残リスク

- `git diff --check` は現在のWindowsサンドボックス読み取り制限により実行できなかった
- ファイル内容確認はNode REPLで実施し、BOM維持と主要文言の反映を確認した

## 次に見るべきこと

- 実運用で「軽微なmerge/Ready/close」の判断が広がりすぎないか
- ユーザー確認不要と判断した理由が記録されているか
- 内部承認がユーザー承認必須事項の代替として扱われていないか
