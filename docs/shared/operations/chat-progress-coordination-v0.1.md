# Chat Progress Coordination Policy v0.1

作成日: 2026-06-13 JST
Owner視点: PdM / PM

## 1. 結論

プロジェクト作業では、各チャットが進捗を確認する運用は必要です。
ただし、毎回重い報告を義務化すると開発速度が落ちるため、必須にするのは次の3点だけです。

- 開始時に `PROGRESS.md` を読む。
- 編集前に担当範囲と衝突リスクを確認する。
- 終了時に成果、確認結果、残タスク、連携事項を残す。

調査だけで成果物や判断が残らない場合は、終了時更新は省略できます。

## 2. 目的

- 同じファイルを別チャットが同時に編集する事故を防ぐ。
- 既存の未コミット変更を上書き・巻き戻ししない。
- DB、secret、production/staging、外部サービスに関する危険操作を事前に止める。
- ユーザーが「どこまで終わっていて、次に何をすればよいか」を短時間で把握できる状態にする。
- PM/PdMの判断が必要な論点を、実装チャットに埋もれさせない。

## 3. 各チャットの必須手順

### 開始時

1. `PROGRESS.md` を読む。
2. 作業テーマに該当するdocsを読む。
3. 実装を伴う場合は `docs/shared/quality/two-pass-task-test-policy-v0.1.md` を読む。
4. 既存の未コミット差分がある場合は、自分の作業範囲と重なるか確認する。
5. 編集前に `PROGRESS.md` のアクティブ作業枠へ担当範囲を書く。

### 作業中

- 予定外のファイルに触る必要が出たら、作業範囲を更新する。
- 他チャットの作業枠と重なる場合は、勝手に進めずユーザーへ確認する。
- DB更新、migration、secret、外部サービス、本番・staging操作は、実行前に安全条件を書く。
- read-only、dry-run、previewで済む確認は先に実行する。

### 終了時

`PROGRESS.md` に以下を残します。

- 実施したこと
- 変更した主なファイル
- 確認したこと
- 未完了のこと
- 次チャットが最初に見るべきファイル
- ユーザー確認が必要なこと

## 4. 連携が必須の条件

| 条件 | 理由 | 必要な対応 |
|---|---|---|
| 同じファイルを触る | 差分衝突や巻き戻しのリスク | 先行チャットの完了・方針を確認 |
| `app/api/`, `lib/`, `prisma/` を触る | 挙動やDB契約への影響 | 影響範囲とテストを明記 |
| `.env*`, `private/`, `secrets/` に関係する | secret漏えいリスク | 値を書かず、必要操作だけ記述 |
| DB write、migration、seed、delete | データ破壊リスク | dry-run、対象件数、rollback可否を確認 |
| production/staging操作 | 公開環境への影響 | Owner確認を明記 |
| package更新 | buildや依存関係への影響 | 目的、影響範囲、lockfile更新を明記 |
| 認証・権限・メール送信 | セキュリティと業務事故リスク | role別確認または送信条件を明記 |
| 要件変更・優先順位変更 | PdM/PM判断が必要 | 実装前に判断事項として分離 |

## 5. PM/PdM判断ゲート

以下は、実装チャットだけで決めず、PM/PdM観点で確認します。

- 何をMVPに含め、何を後回しにするか。
- 既存業務フローを変えるか、現行運用に合わせるか。
- ユーザーに見える文言、権限、公開範囲を変えるか。
- stagingまたは本番でデータ更新を行うか。
- 実メール送信、外部API接続、料金や契約に影響する操作を行うか。
- 仕様の曖昧さを実装側の推測で固定してよいか。

## 6. 完了条件

各チャットは、終了時に次の状態を目指します。

- 作業範囲が明確である。
- 変更ファイルが説明されている。
- テストまたは確認結果が残っている。
- 未完了事項が次のアクションとして読める。
- Owner確認が必要な事項だけが残っている。
- secretや個人情報がdocsに残っていない。

## 7. 他チャットへ送る文面テンプレート

### 作業開始時

```text
このチャットで作業を始める前に、ルートの `PROGRESS.md` を確認してください。
確認後、担当する範囲・触る予定の主なファイル・衝突しそうな既存作業があるかを最初に宣言してください。
実装を伴う場合は `docs/shared/quality/two-pass-task-test-policy-v0.1.md` も確認し、終了時に `PROGRESS.md` へ変更内容・確認結果・残タスクを追記してください。
```

### 既存作業と重なりそうな時

```text
この作業は他チャットの担当範囲または未コミット差分と重なる可能性があります。
先に `PROGRESS.md` のアクティブ作業枠と `git status` を確認し、重なるファイル、進めてよい範囲、待つべき作業を整理してから着手してください。
判断が必要な場合は、実装せずに確認事項として返してください。
```

### 作業終了時

```text
作業終了前に `PROGRESS.md` を更新してください。
最低限、実施内容、変更ファイル、確認結果、未完了、次チャットへの連携事項、Owner確認が必要な事項を残してください。
secret、DB接続URL、パスワード、個人情報は書かないでください。
```

### DB・staging・productionに関係する時

```text
DB更新、migration、staging、本番、外部サービス操作に入る前に、必ず安全条件を明文化してください。
dry-runまたはpreviewで確認できるものは先に実施し、実データ更新が必要な場合は対象範囲、件数、rollback可否、Owner確認事項を分けて報告してください。
```
