# Market / Search / Gmail Recheck

## A/B/C 分類

| 区分 | 意味 | 今回の扱い |
|---|---|---|
| A | 早めに戻す・合わせるべきもの | 既存導線やユーザー期待に直結する表示・検索条件 |
| B | 承認や仕様確認が必要なもの | dashboard API、DB apply、ラベルや抽出仕様の変更 |
| C | 後続でよいもの | 表示 polish、調査記録、運用台帳 |

## Market

| 項目 | 分類 | メモ |
|---|---|---|
| 取得件数を自由入力 | A | 固定選択だけにしない。件数は人間が調整できるようにする |
| 初期値 100 | A | 最初の表示は 100 件を基準にする |
| 直近3か月 | A | 期間プリセットとして確認する |
| 累計 2026-01 から | A | 2026年1月以降の累計を見る軸を残す |
| 単価 5万円刻み | A | 価格帯の比較を細かすぎず粗すぎない粒度にする |
| 戻る導線 | A | 詳細・絞り込み後に元の一覧へ戻れること |
| 表をコンパクトにする | C | 情報量を落とさず、一覧で比較しやすくする |

## Search / Gmail

| 項目 | 分類 | メモ |
|---|---|---|
| AI vs ARI | B | `AI` と `ARI` は別ラベルとして扱い、検索や表示で混同しない |
| `Person.name` 分離 | B | 個人名と会社名・担当者名が混ざらないようにする |
| 会社抽出ラベル | B | Gmail 抽出結果に「会社として抽出した」ことが分かるラベルを付ける |
| fallback | B | 自動抽出できない場合の代替表示・手動確認ルートを残す |
| dashboard API | 承認待ち | dashboard API 変更は影響範囲が広いため、実装前に承認を取る |

## Gmail会社補完

- preview と apply を分ける。
  - preview は「何が変わるかを見るだけ」。DB更新はしない。
  - apply は「DBへ反映する操作」。承認後に実施する。
- apply 前に DB接続先を確認する。接続先のURLやpasswordは docs に書かない。
- rollback は必須。rollback は「反映後に問題があった場合に元へ戻す手順」。
- apply は DB write なので、この docs-only PR では行わない。
