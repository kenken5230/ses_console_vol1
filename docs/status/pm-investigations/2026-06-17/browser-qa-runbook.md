# Browser QA Runbook

## 原則

- 通常ログインのみで確認する。
- cookie、token、auth proxy は使わない。auth proxy は認証を迂回・代理する仕組みのこと。
- secret、password、DB URL は docs やログに書かない。
- DB write、migration、deploy は行わない。

## 共通確認

1. 通常のログイン画面からログインする。
2. Console に重大な error や duplicate key warning がないか見る。
3. 画面上の導線が押せるもの・押せないものとして自然に見えるか確認する。
4. 未実装のものは、通常機能のように見せない。
5. 画面遷移後に戻る導線があるか確認する。

## `/`

- Header nav が見える。
- 市場分析への導線が見える。
- 検索、フィルター、並び替え、ページングが破綻しない。
- 検索履歴は現 main の扱いに従い、sample-only か DB-backed かを混同しない。
- 提案開始は実DB write しない状態なら、その旨が画面上で分かる。

## `/imports`

- 一覧・詳細・戻る導線が破綻しない。
- dry-run と apply の違いが分かる。dry-run は「確認だけ」、apply は「反映」。
- apply が必要な操作は承認なしに行わない。

## `/matches`

- マッチ候補の一覧、レビュー状態、詳細表示が崩れない。
- duplicate key warning が出ない。
- 保存・承認・却下など DB write につながる操作は、承認状態を確認してから扱う。

## `/market-analysis`

- 初期取得件数 100 の期待と実画面の差を確認する。
- 取得件数の自由入力、直近3か月、2026-01からの累計、5万円刻み、戻る導線、コンパクトな表を確認する。
- 詳細・フィルター後に元の比較へ戻れるか見る。
- 表示文言とデータ件数が矛盾しないか確認する。

## 記録形式

- 確認日時。
- route。
- login 方法は「通常ログイン」とだけ書く。認証情報は書かない。
- OK / NG / Blocked。
- Blocked の場合は、何が足りないかを短く書く。
